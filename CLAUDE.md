# Golem ‚Äî Agent Contract

Audience: Claude. Golem is a production expression-evaluation engine: a **thin,
idiomatic wrapper** around [`github.com/expr-lang/expr`](https://github.com/expr-lang/expr)
that lets non-engineers author expressions like `2 + 3 * (x - 1)` which backend
services evaluate millions of times per second ‚Äî fast, safe, and never silently
wrong. A Python binding generated by `gopy` over the same Go core ships as a
tier-2, best-effort layer. A Thaumcraft Golem runs its Seal's instruction
exactly and tirelessly, powered by Vis ‚Äî mirroring this package running an
expression exactly, powered by the `vis` core.

## The product, in one line

Three "by default"s ‚Äî **Fast** (compile-once + bounded LRU cache), **Safe**
(sandboxed function/math allowlist, cost/timeout limits, panic‚Üíerror boundary,
engine-agnostic API), **Loud** (declared schemas turn typos into *compile*
errors, never silent zeros). North star: **zero silently-wrong evaluations in
production.** golem is NOT a new language, NOT Turing-complete, NOT a
rules/workflow engine.

## Behavioral contracts

1. **Thin wrapper ‚Äî do not reinvent the wheel.** expr-lang already provides
   parsing, type-checking (`expr.Env`), custom functions (`expr.Function`),
   builtins/math, `??`/`?.`/`let`, `AllowUndefinedVariables`, and a thread-safe
   reusable `*vm.Program`. golem adds ONLY: a bounded LRU `*vm.Program` cache,
   function/registry ergonomics, an explicit null policy, a panic‚Üíerror boundary,
   cost/timeout guards, and a typed API. If you are writing a parser, lexer,
   type-checker, operator table, or hand-rolled C ABI ‚Äî STOP. A framework already
   does it.
2. **No expr type leaks.** No expr-internal type appears in any public field,
   signature, error message, or Python exception. The API is engine-agnostic so
   the engine stays swappable.
3. **No panic escapes.** Recover at the `safeRun` boundary and return a typed
   error / exception. Always.
4. **YOU MUST NOT fabricate numbers.** The honest-numbers contract is the
   product. Every benchmark/coverage/adopter claim reports REAL measured values
   (`go test -bench=. -benchmem`). Never invent `ns/op`, `allocs/op`, coverage %,
   or adopters. A prose "looks done" verdict is not a PASS ‚Äî technical fact-check
   + buildability are mandatory gates (the hollow-DEPLOY lesson).
5. **Honest dependency licenses.** `expr-lang/expr` = MIT; `hashicorp/golang-lru/v2`
   = **MPL-2.0, NOT MIT** (permissive file-level copyleft, compatible). Never
   mislabel.
6. **Scope guard (v1).** NO Web3 / on-chain registry / Ed25519 / MCP server /
   model-tier logic / Explain Mode in v1 code. Those are roadmap, documented in
   the README, never built into v1.
7. **gopy is the binding path.** Generate the Python extension with `gopy` ‚Äî do
   NOT hand-roll a C ABI. The dynamic path crosses a JSON envelope via `EvalJSON`
   (gopy #254 workaround). If gopy proves intractable, HALT and surface the
   blocker ‚Äî do not silently switch to the CGo `c-shared` + cffi fallback (it is
   ADR-documented as a known trade-off, MUST NOT be implemented in v1).
8. **Authorship = "Enchanter Labs"** on all public artifacts (LICENSE, README,
   docs, PDFs). golem is MIT.

## Honest-numbers contract

The deliverable's credibility is its honest numbers. A verdict (PASS / COMPLETE /
VERIFIED / "production-grade") must carry: the sample size `n`, the sampling
method, and a calibration qualifier. The integration/verify step reports which
gates RAN vs were SKIPPED for missing tooling (e.g. gopy / cibuildwheel absent) ‚Äî
no green-washing. The Python binding is written and reviewed but compiles only
where gopy + a C toolchain exist; say so plainly.

## F-code taxonomy

Self-observed failures are logged with the shared `vis` failure-mode codes (see
`@../vis/packages/core/conduct/failure-modes.md`) so cross-session learning can
aggregate. The ones golem hits most:

- **F01 sycophancy** ‚Äî agreeing with a proposal without an adversarial check
  (counter: doubt-engine).
- **F22 capability-blind substitution** ‚Äî when a named capability (gopy, a
  linter) is absent, recover/escalate/abort; never silently degrade and ship
  under the original bar.
- **F24 substrate-blindness** ‚Äî acting before consuming briefing / MEMORY /
  learnings / precedent.
- **F28 reinvention** ‚Äî authoring a tool/parser/cache that a framework already
  provides (the grading crux for golem specifically).

A prose "DEPLOY"/"looks done" score is NOT production-grade on its own:
technical fact-check (vs primary sources), implementability (does it build?),
and strategy gates are mandatory before any green verdict.

## Acceptance gates

On a clean checkout, all must pass (self-verify; fix and re-run until green):
`go build ./...`; `go vet ./...`; `go test ./...`; `go test -race ./...` (N
goroutines, one shared Program); `go test -bench=. -benchmem` (cached Eval in
tens-of-ns, big speedup vs cold ‚Äî REAL numbers in README + WRITEUP);
`go test -fuzz=FuzzCompile -fuzztime=30s` (no panic escapes); golangci-lint /
staticcheck / govulncheck clean. Python (where gopy + C toolchain exist): `gopy
build` succeeds; cibuildwheel produces wheels; `pip install` + `pytest` pass on a
machine without Go; a parity test asserts identical Go/Python results over a
shared corpus.

## Anti-patterns

- **Reinventing expr.** Writing any parsing/type-checking/operator logic golem's
  dependency already provides.
- **Leaking expr types.** An expr type in a public signature, field, or message.
- **Fabricated benchmarks.** Any `ns/op` not produced by a real `go test -bench`.
- **Silent fallback to a hand-rolled C ABI** instead of HALTing on a gopy blocker.
- **Mislabeling golang-lru/v2 as MIT.** It is MPL-2.0.
- **A prose green verdict without a build + fact-check.** That is hollow-DEPLOY.
